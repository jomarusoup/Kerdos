## 1. 개발 개요

### 1.1 프로젝트명

**ASTP (Autonomous Strategy Trading Platform)** — 미확정, 잠정 코드네임.

### 1.2 개발 목적

기관·투자 전문가의 트레이딩 전략을 기반으로, **AI 에이전트가 사용자의 계좌
잔고를 분석해 최적화된 매매 시나리오를 자율 실행하는 시스템**을 구축한다.

### 1.3 현재 시스템과의 관계

현재 `main`의 Kerdos ATAS는 GPT-4o가 매 틱마다 재량 판단(discretionary)을
내리는 구조다. ASTP는 이를 **검증된 전략 규칙 + AI 보조** 구조로 전환한다.

| 구분 | 현재 ATAS (`main`) | 신규 ASTP (`feat/v2-astp`) |
| --- | --- | --- |
| AI 엔진 | OpenAI GPT-4o | **Azure AI Foundry** |
| 의사결정 | 매 틱 AI 재량 판단 | **검증된 페르소나 규칙 + AI 필터** |
| 리스크 통제 | SL/TP 값 생성만 (미집행) | **전용 리스크 모듈 + 킬스위치** |
| 사용자 맞춤 | 단일 전략 | **계좌 잔고 기반 페르소나 선택** |
| 모니터링 | Streamlit 단순 대시보드 | **Monitor Web (의사결정 근거 포함)** |
| 재현성 | 비결정적 | **규칙 기반 결정성 확보** |

> 현재 `main`은 실거래가 동작하는 운영 코드이므로 **보존**한다.
> 신규 시스템은 별도 브랜치에서 개발하고, 페이퍼 트레이딩 검증 후 머지한다.

## 2. 핵심 기능 및 서비스 구성

### 2.1 멀티-페르소나 전략 라이브러리

기관·투자 전문가의 투자 스타일을 **프롬프트로 변환**하여 사용자가 선택할 수
있는 전략 옵션으로 제공한다.

- 각 페르소나 = `{투자 철학 + 진입/청산 규칙 + 리스크 성향}`의 구조화된 정의
- 예시 페르소나: 가치투자형, 모멘텀 추종형, 역추세 매매형, 변동성 돌파형
- 사용자는 자신의 계좌 상황·성향에 맞는 페르소나를 선택
- 각 페르소나는 **백테스트로 사전 검증**하여 양(+)의 기대값을 확인한 뒤 노출

### 2.2 트레이딩 결정 에이전트

외부 금융 데이터와 사용자 계좌 잔고 데이터를 분석해 매매를 결정한다.

**입력**
- 시세 (OHLCV, 오더북)
- 거래원(매매 주체) 정보
- 기관 수급 동향
- 사용자 계좌 잔고 (보유 자산·현금 비중)

**출력**
- 매매 신호 (`buy` / `sell` / `hold`) + 포지션 비중 + 근거 + 신뢰도
- SL/TP 수준 (집행까지 연결)

### 2.3 자율 매매 시스템

결정 에이전트의 신호를 기반으로 **실시간 매매를 집행하는 인터페이스**.

- 신호 → 리스크 필터 통과 → 주문 집행
- SL/TP 집행 루프 (가격 감시 → 자동 손절/익절)
- 주문 체결·실패 처리 및 재시도 로직

## 3. 시스템 구현 아키텍처

### 3.1 디렉토리 구조

```
feat/v2-astp/
│
├── core/
│   ├── strategies/          # 멀티-페르소나 전략 라이브러리
│   │   ├── base.py          #   페르소나 추상 인터페이스
│   │   └── personas/        #   개별 전략 프리셋 (프롬프트 + 규칙)
│   │
│   ├── agent/               # 트레이딩 결정 에이전트
│   │   ├── foundry.py       #   Azure AI Foundry 연동
│   │   ├── decision.py      #   신호 생성·평가
│   │   └── evaluator.py     #   매매 결과 평가
│   │
│   ├── pipeline/            # 외부 데이터 수집·정제 파이프라인
│   │   ├── market.py        #   시세·오더북
│   │   ├── supply.py        #   기관 수급·거래원
│   │   └── account.py       #   계좌 잔고 수집
│   │
│   ├── trading/             # 자율 매매 실행 인터페이스
│   │   ├── executor.py      #   주문 집행
│   │   └── monitor_loop.py  #   SL/TP 집행 루프
│   │
│   ├── risk/                # 리스크 프레임워크
│   │   ├── sizing.py        #   포지션 사이징
│   │   └── guard.py         #   일일 손실 한도·MDD 킬스위치
│   │
│   ├── backtest/            # 백테스트 엔진
│   │   ├── engine.py        #   과거 OHLCV 시뮬레이션
│   │   └── metrics.py       #   승률·MDD·샤프·손익비
│   │
│   └── db/                  # 데이터 저장소
│       └── schema.py        #   전략 이력·잔고·매매 결과
│
├── web/                     # Monitor Web
│   └── (전략 현황·의사결정 근거·리포트 이력 시각화)
│
└── config/                  # Azure AI Foundry 설정·페르소나 프리셋
```

### 3.2 구성 요소별 책임

| 영역 | 기술 | 책임 |
| --- | --- | --- |
| **AI** | Azure AI Foundry | 투자 전략 분석, 매매 결과 평가 |
| **데이터 처리** | 자체 파이프라인 | 외부 데이터 + 잔고 데이터 수집·정제 |
| **모니터링** | Monitor Web | 전략 선택·잔고 상태·의사결정 근거·리포트 이력 시각화 |
| **DB** | PostgreSQL | 전략 선택 이력·잔고·매매 결과 저장 (맞춤형 데이터 확보) |

## 4. 단계별 개발 계획

| Phase | 목표 | 주요 산출물 | 우선순위 |
| --- | --- | --- | --- |
| **1** | 기반 인프라 | Azure AI Foundry 연동(`agent/foundry.py`), DB 스키마(`db/schema.py`), 데이터 파이프라인 골격 | 최상 |
| **2** | 백테스트 엔진 | `backtest/engine.py`, `backtest/metrics.py` (수수료·슬리피지 반영) | 최상 |
| **3** | 전략 라이브러리 | 멀티-페르소나 코드화(`strategies/personas/`), 백테스트 유효성 검증 | 상 |
| **4** | 결정 에이전트 | 외부 데이터 + 잔고 통합 분석(`agent/decision.py`), 매매 신호 생성 | 상 |
| **5** | 리스크 + 자율 매매 | `risk/`, SL/TP 집행 루프(`trading/monitor_loop.py`), 주문 집행 | 상 |
| **6** | Monitor Web | 전략 현황·의사결정 근거·계좌 상태·리포트 이력 시각화 | 중 |
| **7** | 페이퍼 트레이딩 | N주 모의 검증 → 통과 시 `main` 머지 | 최상 |

> **착수 순서 권장**: Phase 1(인프라) → Phase 2(백테스트). 백테스트 엔진이
> 없으면 이후 모든 전략·리스크 변경이 추측에 머문다. 검증 토대를 먼저 확보한다.

## 5. 핵심 철학

```
[현재 ATAS]  매 틱마다 "AI야 어떻게 할까?" → 그대로 실행
[신규 ASTP]  검증된 페르소나 규칙으로 신호 생성
             → 리스크 필터 통과 → 자율 집행
             → AI는 "이 신호를 거부할 이유가 있나?"라는 보조·평가 역할
```

AI를 버리는 것이 아니라, **검증 불가능한 의사결정자**에서
**검증된 규칙 위의 필터·맥락 제공자·결과 평가자**로 역할을 이동시킨다.
